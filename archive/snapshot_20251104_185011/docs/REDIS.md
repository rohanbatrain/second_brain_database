Redis configuration and cloud fallback

This project uses Redis for short-lived session/storage and abuse tracking.
The application will try to connect to a local Redis instance first (using
`REDIS_HOST`, `REDIS_PORT`, `REDIS_DB`). If that fails at startup it will
attempt to connect using the `REDIS_STORAGE_URI` configuration (for example
Redis Cloud / Heroku-style URL). If both attempts fail the application will
exit (by design) so the deployment fails fast.

Where to configure
- `.sbd_config.json` (generated by `generate_config.sh`) or
- Environment variables (preferred for production) - see `.env.production.example`.

Environment variables used
- REDIS_HOST (e.g. 127.0.0.1)
- REDIS_PORT (e.g. 6379)
- REDIS_DB (e.g. 0)
- REDIS_STORAGE_URI (e.g. redis://:password@host:port/0) -- overrides local host when present
 - REDIS_USERNAME (optional)
 - REDIS_PASSWORD (optional)

Example (using python redis client)

```py
import redis

# Direct construction using host/port/username/password (blocking client)
r = redis.Redis(
  host='redis-11991.c13.us-east-1-3.ec2.redns.redis-cloud.com',
  port=11991,
  decode_responses=True,
  username='default',
  password='REDACTED_PASSWORD',
)
assert r.ping()
print(r.get('foo'))
```

Notes
- The code will prefer a local Redis first. Use `REDIS_STORAGE_URI` for managed
  Redis (Redis Cloud) and include credentials in the URI.
- Keep secrets out of git. Use environment variables or your secret manager.
- See `.env.production.example` for a production env example.
